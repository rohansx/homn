## Why this number is NOT a brain-architecture decision

This run is a **pipeline validation on real capture**, not a valid Phase-0 gate verdict. The
sample is too thin and too non-representative to decide agidb-vs-ctxgraph:

1. **~45 minutes of capture.** The plan calls for 5–7 working days.
2. **No real work activity on screen.** All 12 OCR frames are an identical static Konsole
   "File" menu (`File / New Window / New Tab / Built-in / Default profile / Clone Tab / Save
   Output As / Auto Save Output As`). The user was not actively switching apps or producing
   varied screen content during this window.
3. **The audio is background media, not the user's speech.** 168 transcriptions, all from the
   mic (`alsa_input`) picking up a YouTube video playing through speakers — motivational/lyrics
   content ("if you want to do something like this, you can do it", "use a free machine for a
   long time", "make a balance between the two step", "all our music"). Zero mentions of the
   actual work (homn, capture, eval, recall, brain, commit, agent, people, meetings).
4. **The questions are grounded in the captured content**, so they're answerable by
   substring/keyword recall — which agidb handles trivially. A recall@3 of 76.7% on
   distinctive-phrase retrieval says little about recall over a real week of mixed, noisy,
   multi-app activity.

The number crosses 70% *because the task is easy*, not because the brain is proven. The
verdict is **deferred** until a real capture week (see `research.md` R1 outcome).

## What this run DOES prove

- The full Phase-0 pipeline runs end-to-end against the **live** screenpipe capture DB, not a
  fixture: `homn eval ingest ~/.screenpipe/db.sqlite` → 194 rows read, 44 chunks stored; the
  introspection handles the real v2 schema (`frames.full_text` + `audio_transcriptions`).
- `homn eval run` scores, prints the verdict table, and emits `--json` — the CLI surface is
  real and the gate machinery (score → `gate_verdict` → report) works on real data.
- agidb recall is wired and returns ranked hits with text; the `AgidbRecaller` bridge
  (sync `Recaller` → async `recall_cue`) works inside the CLI's tokio runtime.

## Blockers to a meaningful gate (found this run)

1. **Ambient audio is background-media noise.** The mic transcribes a YouTube video, not the
   user's speech. Fix: disable screenpipe's audio capture; rely on convox-voice (push-to-talk)
   as the intentional-speech source. (Applied: see the screenpipe service unit.)
2. **convox-voice does not persist transcriptions.** Only 4 old log lines (Jul 9, also noise)
   carry `text="..."`; the current faster_whisper backend emits none. The DictationPipe source
   (T026) and `eval ingest` therefore have no intentional-speech feed yet. Fix: convox-voice
   must append each final transcription (with a timestamp) to a JSONL log the eval can read.
3. **The capture week has not actually started.** ~45 min of a static menu is not a working
   week. The user must work at the machine for 5–7 days with the fixes above in place.

Once those are resolved, re-author a question set from the real week and re-run; the resulting
recall@3 is the number that decides the brain branch.
